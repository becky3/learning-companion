# features テンプレート

`features/` カテゴリの仕様書に適用する。ユーザーが直接触る・意識するプロダクト機能の定義。

## セクション一覧

| セクション | 義務 |
|---|---|
| 概要 | 必須 |
| 背景 | 必須 |
| 制約 | 任意 |
| 操作一覧 | 必須 |
| 各操作の仕様 | 必須 |
| エッジケース | 任意 |
| コンポーネント構成 | 必須 |
| 関連ドキュメント | 任意 |

## 概要 [必須]

この仕様が「何を」定義するのかを簡潔に記載する。

- 対象の目的・役割を簡潔に
- スコープ（何を含み、何を含まないか）— 必要な場合のみ

**記載しない**:

- 技術的な詳細
- 経緯・動機

## 背景 [必須]

この仕様が存在する理由を記載する。

- 解決する課題・動機
- 誰にとってどう有益か

## 制約 [任意]

振る舞いの制約・既知の制限事項・前提条件を記載する。

- 不変条件（「〜してはならない」「〜の場合は〜する」）
- 既知の制限事項・環境依存の注意点

How の追加が必要な場合は HTML コメントで理由を記録する。

## 操作一覧 [必須]

この機能が提供するコマンド・操作の全体像を記載する。

- 操作名とその概要の一覧（テーブル形式）
- 各操作のトリガー方法（Slack コマンド、定時実行、API 呼び出し等）

## 各操作の仕様 [必須]

操作ごとに h3 サブセクションを作成し、個別の振る舞いを定義する。

- トリガー（何がこの操作を開始するか）
- 振る舞い（処理の概要・条件分岐）
- 出力（結果・応答・副作用）

## エッジケース [任意]

通常フローから外れるケースの振る舞いを記載する。

**例**:

- 異常入力・境界値での振る舞い
- タイムアウト・接続エラー時の挙動
- 複数操作が競合した場合の優先ルール

## コンポーネント構成 [必須]

この機能を構成するコンポーネントの関係とデータの流れを定義する。

## 関連ドキュメント [任意]

関連する仕様書や外部ドキュメントへのリンクを記載する。
