# ドキュメント記述ルール

詳細は `docs/specs/style-guide.md` を参照。

## 基本方針

- **What（何をするか）/ Why（なぜそうするか）を中心に書く**。How（どう実装するか）は原則書かない
- ドキュメント内の図表は **mermaid 形式**を使用する（ASCII図表は不可）

## 仕様書に書かないもの

- 疑似コード・クラス定義・メソッド実装
- 具体的な設定値（ポート番号・URL・数値定数・テーブル名・ユーザー名・モデル名）
- 受け入れ条件（AC は Issue に記載）
- 変更履歴・実装ステータス・PR番号（Git/GitHub で管理）
- 実験データ・コスト見積もり（Issue に記録）
- テスト方針（Issue に記載）

## mermaid の注意事項

- **mermaid ブロック内の `#` 禁止**: GitHub の Mermaid レンダラーがパースエラーになる（CSS カラーコード `#xxx`/`#xxxxxx` は例外）
- `#` が必要な箇所は文脈に応じて置換する（例: `feature/chat-#1` → `feature/chat-N1`）

## TODO 形式

未修正の既知制限・将来対応予定の事項:

```markdown
> **TODO:#Issue番号** 説明
```

- 対応 Issue が存在することが前提（Issue 番号必須）
- Issue 解決後に TODO を削除または本文に反映する
