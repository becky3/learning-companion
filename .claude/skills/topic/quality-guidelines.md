# 執筆品質ガイドライン

記事生成時に必ず以下を適用する。Zenn/Qiita のトレンド調査から導出した指針。

## タイトル

- 「痛み」か「意外性」を含める（トレンド記事の共通特徴）
- 40-60文字を目安
- 効果的なパターン:
  - `{X}したら{Y}だった話` — 体験ベース
  - `{X}で{Y}を自動化した` — 成果ベース
  - `{X}の{N}つのポイント` — リスト系
  - `なぜ{X}で{Y}なのか？` — 疑問・問題提起系
  - `私が実際に使っている{X}だけの{Y}` — 限定・選別系
- NG: 抽象的すぎるタイトル（「クロスプラットフォーム対応について」等）

## 構成の原則

- **1記事1テーマ**: テーマがブレたら記事を分割する。詰め込みすぎない
- **結論ファースト**: 記事全体でもセクション内でも、結論・要点を先に書く。詳細は後に展開
- **前提の明示**: 対象読者、環境・バージョン、前提知識を導入部で明確にする
- **流し読み対応**: 見出しだけで記事の大意が伝わる構造にする。太字で各段落の要点を強調

## 文字数

- 目標: 2,000〜6,000字（Zenn で Like が集まりやすいレンジ）
- 短すぎる場合: 具体例、代替アプローチ、背景の深掘りで補強
- 長すぎる場合: セクションの分割やシリーズ化を提案
- 注: 文字数より内容の質が重要。無理に引き延ばさず、簡潔さを優先する

## 文体

- です/ます調 + 適度な口語（堅すぎない、砕けすぎない）
- 開発者本人の体験として書く（一人称視点）
- `:::message` で重要な注意点を強調
- 不要なヘッジ表現（「かもしれません」「と思われます」）を避ける

## 一次情報の重視

- 公式ドキュメントの焼き直しではなく「やってみたらこうだった」を書く
- エラー解決の試行錯誤プロセス、技術導入の成功・失敗事例に価値がある
- 未確認の内容は「ドキュメントによると」「一般的には」と正直に注記する

## 内容の正確性

- **未検証の機能を書かない**: 仕様書に定義されていても、実際に動作確認していない機能は記事に含めない
- **安全性・動作の表現は正確に**: 「即座に停止」「必ず検出される」等の断定的な安全表現は、実際の挙動と一致しているか厳密に確認する（例: 「次のステップから停止」と「即座に停止」は異なる）
- **「最も」「必ず」「絶対」等の最上級・断定表現を避ける**: 比較対象を網羅的に検証していない限り使わない。「〜のほうが安定する」「〜と判断した」のように、自分の経験に基づく表現にする
- **実際に起きないシナリオを書かない**: 「Xが失敗したらYで停止」のような記述は、Xが実際に失敗するパスが存在するか確認する
- **スコープを守る**: 記事のテーマに直接関係しない情報は含めない。「あると便利かも」レベルでは入れない
- **コスト・料金情報は慎重に**: ユーザーの契約状況で変動する情報は、具体的な金額を避けるか、前提条件を明示する
- **用語は初出で定義する**: 専門用語・略語・英語の技術用語は初出箇所で日本語の補足を括弧書きで入れる（例: `Fine-grained PAT（リポジトリ単位で権限を絞れる個人アクセストークン）`）。記事内で独自に使う用語（例: 「禁止パターン」）も同様

## 導入の書き方

- **推奨: 共感型の問いかけ**（最多・最効果的）
  - 「〜ということはありませんか」「〜で困ったことはありませんか」
- **代替: 対比・逆張り型**
  - 「〜と思いきや」「〜が定番ですが、実は」
- **避ける**: 自己紹介から始める導入（フォロワーベースがない場合は逆効果）

## まとめの書き方

- 箇条書きで要点を整理
- 展望や応用可能性を1-2文で添える
- 「読んでくれてありがとう」等の定型文は不要

## 図表・コード

- **mermaid**: アーキテクチャ図、フローチャート、シーケンス図
- **テーブル**: 比較、条件一覧、設計判断の根拠
  - テーブル内の情報が本文と重複する場合、テーブルは簡潔にして「（X 節参照）」で本文を参照させる
  - 列名・セル内容は読者に伝わる平易な表現にする（技術的な内部名だけでは不十分。例: `statusCheckRollup` → 「CI が全て成功しているか」）
- **コードブロック**: ファイル名注釈付き（` ```python:process_guard.py `）

## 画像・スクリーンショット

- **画像はエンゲージメントに直結する**: Zenn の人気記事（いいね100超）は 4 枚以上の画像を使用している傾向がある。特に GitHub UI のスクリーンショットが効果的
- **プレースホルダー形式**: blockquote を使用（Zenn プレビューで視認できる）

  ```markdown
  > 📸 **TODO** — 画像の説明
  ```

- **`{{...}}` 形式は使わない**: プレビューで目立たず、公開時に差し替え忘れるリスクがある
- **1 プレースホルダ = 1 画像**: 複数の画像をまとめて書かない。テキストの流れの中に 1 枚ずつ配置する
- 概念図は mermaid で作成し、スクリーンショットは「動いている様子」「実際の UI 画面」に使う
- 1 記事あたり 3〜5 箇所程度を目安に、再現可能なものだけ配置する（過去の状態で再現不能なスクショは入れない）

## topics（タグ）の選定

- **2〜5個**を目安に選定する
- **必ず1つは人気トピックを含める**（Python, AI, TypeScript, GitHub Actions, Claude Code 等）
- **記事の具体的な技術を1〜2個含める**（フレームワーク名、ツール名等）
- タグはZennの既存トピックから選ぶ（`https://zenn.dev/topics` で確認）
  - 存在しないタグを作らない。フォロワーがいないため発見されない
- このプロジェクトでよく使うタグ例:
  - AI支援開発系: `ClaudeCode`, `生成AI`, `AI`
  - CI/CD系: `GitHubActions`, `GitHub`, `CI`
  - 言語/環境系: `Python`, `個人開発`

## AI 支援の表記

記事末尾に以下を配置（任意。公開前にユーザーが判断）:

```markdown
:::message
本記事は開発ジャーナルを元に AI の支援を受けて執筆し、人間がレビュー・編集しています。
:::
```
